# CNC 与汽车零部件行业 AI 排产智能体开发文档

## 1. 文档目标

本文档用于指导面向 CNC 机加工与汽车零部件制造场景的 AI 排产智能体产品设计、技术架构、MVP 开发与商业化验证。

产品定位不是单纯聊天机器人，而是以有限产能排产为核心、以智能体交互为入口、以优化引擎和制造数据模型为底座的排产决策系统。

核心目标：

- 缩短计划员排产、插单、重排时间。
- 提高订单准交率与瓶颈设备利用率。
- 降低换型、等待、在制品与人工协调成本。
- 将计划员经验沉淀为可复用、可解释、可审计的排产规则。
- 支持从“人确认”逐步演进到“局部自动排产”。

## 2. 目标行业与核心场景

### 2.1 CNC 机加工行业

CNC 机加工典型特征：

- 多品种、小批量、中短交期。
- 工艺路线包含车、铣、磨、钻、热处理、表面处理、检验等多工序。
- 设备能力差异明显，同类设备也存在精度、行程、刀库、夹具、控制系统差异。
- 换刀、换夹具、首件确认、调机时间对产能影响大。
- 现场插单、返工、设备故障、刀具缺失、外协延迟频繁。
- 很多计划逻辑依赖资深计划员和车间主任经验。

优先解决场景：

- 多工序作业车间有限产能排产。
- 急单插入后的快速重排。
- 瓶颈设备负荷分析。
- 设备故障后的影响评估与替代设备推荐。
- 外协工序与内部工序协同。
- 关键订单交期风险预警。

### 2.2 汽车零部件行业

汽车零部件典型特征：

- 客户交付要求严格，JIT/JIS 交付压力高。
- 订单来源包括预测、滚动计划、客户拉动、售后备件。
- BOM 和工艺路线较稳定，但变更、替代料、版本管理要求高。
- 生产过程可能包含冲压、机加工、热处理、焊接、装配、涂装、检测、包装。
- 模具、夹具、检具、产线节拍、换型窗口影响明显。
- 质量追溯、批次、客户优先级、物流窗口是关键约束。

优先解决场景：

- 客户订单和预测混合驱动的滚动排产。
- 多产线、多设备、多模具协同排产。
- 换型优化与合批生产。
- 缺料、模具异常、设备停机后的动态重排。
- 关键客户订单保交付。
- 产能缺口、加班、外协、转线方案对比。

## 3. 产品定位

### 3.1 产品名称建议

AI Dynamic Scheduling Agent，中文可命名为“AI 动态排产智能体”。

### 3.2 用户角色

- 生产计划员：主要使用者，负责生成和调整排产计划。
- 车间主管：确认计划可执行性，处理现场异常。
- 生产经理：查看产能、交付风险、效率指标。
- 物控/采购：查看缺料对排产的影响。
- 工艺工程师：维护工艺路线、标准工时、替代设备。
- IT/MES/ERP 管理员：负责系统集成、权限、主数据。

### 3.3 核心价值主张

面向 CNC 和汽车零部件企业，提供一个可解释、可重排、可协同的 AI 排产智能体。系统能够在订单、设备、工艺、物料、班次、模具、人员等约束下生成可执行排产方案，并通过自然语言解释计划结果、识别风险、推荐调整动作。

## 4. 功能范围

### 4.1 MVP 功能

MVP 应聚焦一个工厂或一个车间，先完成可演示、可试点、可量化的闭环。

基础数据管理：

- 订单导入。
- 工艺路线导入。
- 设备与产线能力维护。
- 班次日历维护。
- 标准工时维护。
- 物料齐套状态导入。
- 模具、夹具、检具资源维护。

排产功能：

- 有限产能排产。
- 多工序前后关系约束。
- 设备能力匹配。
- 班次和停机时间约束。
- 急单插入重排。
- 设备故障重排。
- 缺料影响重排。
- 方案对比：交期优先、效率优先、少换型优先。

智能体功能：

- 自然语言查询排产结果。
- 自然语言触发重排任务。
- 自动解释延期原因。
- 自动识别瓶颈设备。
- 自动生成交期风险清单。
- 自动生成计划员日报。

可视化功能：

- 订单维度甘特图。
- 设备维度甘特图。
- 设备负荷图。
- 订单交期风险列表。
- 缺料影响列表。
- 方案 KPI 对比面板。

导出与集成：

- Excel 导入导出。
- CSV/API 数据接口。
- 派工单导出。
- MES/ERP 对接预留接口。

### 4.2 非 MVP 功能

以下能力建议放到第二阶段或行业增强版本：

- 多工厂协同排产。
- 多级供应链联动。
- 自动采购建议。
- 设备实时数据 IoT 闭环。
- 质量异常自动追溯。
- 复杂成本优化。
- 强化学习调度。
- 完全无人审批自动下发。

## 5. 智能体能力设计

### 5.1 智能体职责边界

大模型智能体不直接“凭感觉排产”，而是负责理解、编排、解释和协同。真正的排产计算由优化引擎、规则引擎和仿真模块完成。

智能体负责：

- 理解用户意图。
- 补全排产参数。
- 调用排产优化工具。
- 调用数据查询工具。
- 解释排产结果。
- 生成方案对比。
- 识别风险与建议动作。
- 生成日报、周报、异常报告。

智能体不负责：

- 直接编造设备能力。
- 绕过约束强行生成不可执行计划。
- 未经确认自动覆盖正式生产计划。
- 在缺少关键数据时输出确定性结论。

### 5.2 典型对话任务

订单查询：

> 查询订单 SO-202605001 为什么延期。

智能体应返回：

- 当前计划完工时间。
- 客户要求交期。
- 延期时长。
- 关键阻塞工序。
- 阻塞资源。
- 可选改善方案。

插单重排：

> 客户 A 的订单今晚必须先做，帮我评估影响。

智能体应执行：

- 识别订单。
- 检查物料、工艺、设备、模具。
- 创建临时约束。
- 运行重排。
- 对比原计划与新计划。
- 输出受影响订单、延期风险、加班建议。

设备故障：

> CNC-08 今天 14 点到明天 10 点停机，重新排一下。

智能体应执行：

- 写入设备不可用窗口。
- 找到受影响工序。
- 搜索替代设备。
- 运行局部重排。
- 输出替代方案。

方案对比：

> 给我三套方案：保交期、少换型、少加班。

智能体应执行：

- 调用多目标排产。
- 计算 KPI。
- 解释每套方案的取舍。
- 推荐默认方案。

### 5.3 智能体工具集

建议将智能体工具定义为后端可审计 API：

- `query_orders(filters)`
- `query_inventory(material_ids)`
- `query_work_centers(filters)`
- `query_machine_calendar(machine_ids, time_range)`
- `query_routing(part_id)`
- `create_scheduling_scenario(params)`
- `run_scheduler(scenario_id, objective)`
- `compare_schedules(base_schedule_id, candidate_schedule_ids)`
- `explain_delay(order_id, schedule_id)`
- `detect_bottlenecks(schedule_id)`
- `export_dispatch_list(schedule_id)`
- `submit_schedule_for_approval(schedule_id)`

每次工具调用需要记录：

- 用户身份。
- 输入参数。
- 数据版本。
- 约束版本。
- 排产结果版本。
- 智能体解释内容。
- 用户确认动作。

## 7. 排产算法设计

### 7.1 问题建模

离散制造排产可抽象为带资源约束的 Job Shop Scheduling Problem，实际场景通常还包含：

- 多资源约束。
- 替代设备。
- 批量拆分与合并。
- 序列相关换型时间。
- 物料可用时间。
- 班次日历。
- 维护停机。
- 人员资质。
- 外协周期。
- 多目标优化。

### 7.2 推荐技术路线

第一阶段：

- 使用 OR-Tools CP-SAT 作为核心约束求解器。
- 对大规模问题使用启发式预排序。
- 用规则引擎处理业务优先级与硬约束。
- 用滚动窗口降低计算复杂度。

第二阶段：

- 引入混合启发式算法，如遗传算法、禁忌搜索、模拟退火。
- 加入仿真评估模块。
- 支持多目标 Pareto 方案生成。

第三阶段：

- 基于历史排产与执行数据训练调参模型。
- 学习不同计划员的偏好。
- 对局部调度策略做推荐。

### 7.3 目标函数

保交期目标：

- 最小化延期订单数。
- 最小化总延期时长。
- 最大化关键客户订单准时率。

效率目标：

- 最大化瓶颈设备利用率。
- 最小化设备空闲时间。
- 最小化总完工时间。

换型目标：

- 最小化换刀、换模、换夹具次数。
- 最大化同类零件连续加工。

成本目标：

- 最小化加班时长。
- 最小化外协成本。
- 最小化在制品等待。

稳定性目标：

- 最小化对已下发计划的扰动。
- 优先保持今日计划稳定。
- 对未来计划允许更大调整。

### 7.4 硬约束与软约束

硬约束：

- 工序前后关系必须满足。
- 设备不能同时加工两个任务。
- 工序必须分配到具备能力的资源。
- 物料未齐套不能开工。
- 设备停机窗口不可排产。
- 班次外不可排产，除非允许加班。
- 模具、夹具、检具不可超占用。

软约束：

- 尽量减少延期。
- 尽量减少换型。
- 尽量减少加班。
- 尽量减少计划扰动。
- 尽量优先关键客户。
- 尽量合批生产。

## 8. 系统架构

### 8.1 总体架构

建议采用模块化架构：

- 前端工作台。
- 智能体服务。
- 排产优化服务。
- 规则引擎服务。
- 数据接入服务。
- 排产数据仓库。
- 集成适配层。
- 审计与权限服务。

### 8.2 前端工作台

主要页面：

- 排产首页：今日风险、待处理异常、当前方案状态。
- 甘特图：按设备、订单、工序查看计划。
- 订单风险页：延期、缺料、瓶颈、插单影响。
- 方案对比页：多方案 KPI 对比。
- 智能体对话页：查询、解释、重排、报告生成。
- 基础数据页：设备、工艺、班次、工装、物料。
- 审批页：排产方案确认、下发、回滚。

### 8.3 后端服务

智能体服务：

- 意图识别。
- 工具调用编排。
- 上下文管理。
- 权限检查。
- 结果解释。

排产服务：

- 场景创建。
- 约束构建。
- 求解任务调度。
- 方案版本管理。
- KPI 计算。

数据服务：

- 主数据管理。
- 计划数据管理。
- 执行数据同步。
- 数据质量校验。

集成服务：

- ERP 订单接口。
- MES 工单接口。
- WMS 库存接口。
- PLM/BOM 接口。
- IoT/设备状态接口。

## 9. 技术栈建议

前端：

- React 或 Vue。
- Gantt 图组件可选择 dhtmlxGantt、Bryntum Gantt、AntV G6/X6 或自研 Canvas。
- 状态管理使用 Zustand、Pinia 或 Redux Toolkit。

后端：

- Python FastAPI 或 Java Spring Boot。
- 排产算法优先 Python，便于使用 OR-Tools、NumPy、Pandas。
- 业务服务可用 Java 或 Python，根据团队能力选择。

算法：

- OR-Tools CP-SAT。
- PuLP/Pyomo 或商业 MILP 求解器。
- 自研启发式算法。
- Pandas 做数据预处理。

智能体：

- LLM API 或私有化大模型。
- Tool Calling。
- RAG 检索企业工艺规则、计划规则、异常处理手册。
- 结构化输出 JSON Schema。

数据：

- PostgreSQL。
- Redis。
- 对象存储保存导入文件和方案快照。
- 向量数据库可选，用于知识库检索。

部署：

- Docker Compose 起步。
- Kubernetes 用于中大型客户。
- 支持私有化部署。

## 10. MVP 开发计划

### 10.1 第 1 阶段：基础排产闭环

周期建议：6 到 8 周。

交付内容：

- Excel/CSV 数据导入。
- 订单、工艺路线、设备、班次基础模型。
- OR-Tools 基础有限产能排产。
- 设备维度甘特图。
- 订单交期风险列表。
- 方案导出。

验收标准：

- 能对真实或半真实数据生成可执行计划。
- 支持至少 100 个订单、500 道工序、30 台设备的排产。
- 能展示延期订单和设备负荷。

### 10.2 第 2 阶段：动态重排与智能体解释

周期建议：6 到 8 周。

交付内容：

- 急单插入。
- 设备停机。
- 缺料影响。
- 局部重排。
- 智能体自然语言查询。
- 延期原因解释。
- 方案 KPI 对比。

验收标准：

- 插单后 5 分钟内生成新方案。
- 能解释主要延期原因。
- 能展示受影响订单清单。
- 能比较原方案与新方案。

### 10.3 第 3 阶段：行业增强与系统集成

周期建议：8 到 12 周。

交付内容：

- CNC 行业字段增强。
- 汽车零部件行业字段增强。
- ERP/MES API 对接。
- 审批和下发流程。
- 权限与审计。
- 计划员日报和生产风险日报。

验收标准：

- 可在一个试点车间连续运行 4 周。
- 计划员可基于系统结果调整并下发计划。
- 关键排产规则可配置。
- 数据同步稳定。

## 11. 关键 KPI

业务 KPI：

- 订单准交率 OTD。
- 平均延期时长。
- 瓶颈设备利用率。
- 换型次数。
- 加班时长。
- 在制品等待时间。
- 计划员排产耗时。
- 插单响应时间。
- 计划变更次数。

系统 KPI：

- 排产计算耗时。
- 可行解生成率。
- 数据导入成功率。
- 智能体工具调用成功率。
- 解释命中率。
- 用户采纳率。
- 方案回滚次数。

## 12. 数据质量要求

最小必需数据：

- 订单号、零件号、数量、交期。
- 工艺路线和工序顺序。
- 每道工序可用设备。
- 标准工时或节拍。
- 设备班次日历。
- 当前在制状态。

强烈建议补充：

- 准备时间。
- 换型规则。
- 模具/夹具/刀具资源。
- 物料齐套状态。
- 设备维护计划。
- 外协周期。
- 历史实际工时。

数据治理规则：

- 缺少关键字段时不允许自动排产。
- 标准工时异常时提示人工确认。
- 工艺路线版本需要留痕。
- 排产使用的数据快照必须可追溯。

## 13. 风险与应对

数据不准：

- 先做数据质量评分。
- 输出不可排原因。
- 支持计划员手动修正并回写。

现场规则复杂：

- 采用规则模板加客户配置。
- 将人工调整记录转成规则候选。
- 定期复盘计划变更原因。

排产结果不被信任：

- 强化可解释性。
- 每个延期、换型、设备选择都给出原因。
- 提供原计划与新计划差异。

计算规模过大：

- 使用滚动窗口。
- 固定近期已下发计划。
- 对远期计划做粗排。
- 对瓶颈资源重点优化。

实施周期长：

- 先 Excel POC。
- 再单车间试点。
- 最后 ERP/MES 集成。

## 14. 商业化路径

### 14.1 试点切入

推荐 POC 口径：

- 选择 1 个车间。
- 导入 1 到 3 个月订单和工艺数据。
- 使用 2 到 4 周生产窗口验证。
- 对比人工排产和系统排产。
- 评估准交率、换型、设备利用率、计划耗时。

### 14.2 产品套餐

基础版：

- Excel 导入。
- 单车间排产。
- 甘特图。
- 基础智能体解释。

专业版：

- 动态重排。
- 多目标方案。
- ERP/MES 对接。
- 权限审批。
- 行业规则模板。

企业版：

- 多车间、多工厂。
- 私有化部署。
- 高级优化算法。
- IoT 实时闭环。
- 定制报表与 SLA。

### 14.3 首批客户画像

优先选择：

- 年产值 1 亿到 20 亿的离散制造企业。
- 已有 ERP 或 MES，但排产仍主要靠 Excel。
- 订单交期压力明显。
- 产品多品种、小批量。
- 管理层有数字化投入意愿。
- 现场愿意配合数据梳理。

## 15. 推荐落地顺序

第一步：选择一个行业样板，例如 CNC 机加工。

第二步：用 Excel 数据完成单车间排产 MVP。

第三步：补齐智能体解释、插单重排、设备故障重排。

第四步：拿真实客户数据做 POC。

第五步：沉淀行业规则模板。

第六步：扩展到汽车零部件场景。

第七步：做 ERP/MES 深度集成。

第八步：推出私有化和企业版。

## 16. 结论

CNC 和汽车零部件都是适合 AI 排产智能体切入的高价值场景。CNC 更适合作为 MVP 起点，因为痛点集中、流程相对可控、Excel 排产普遍、POC 周期较短。汽车零部件更适合作为增强行业版本，因为客户交付压力大、规则复杂、客单价更高，但集成和实施难度也更高。

建议产品从“计划员副驾”开始，先解决排产生成、异常重排、结果解释和方案对比，再逐步演进到半自动排产和局部自动下发。技术实现上，应坚持“优化引擎负责计算，智能体负责理解和解释，规则引擎负责行业约束，数据模型负责可执行性”的架构边界。
